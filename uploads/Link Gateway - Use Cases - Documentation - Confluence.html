Introduction

This document follows on from Link Gateway ISO Installation, Link Gateway Initial Configuration, Link Gateway - Configuration via Platform.  Its purpose is to describe typical installation scenarios and use cases for Link Gateway.

 

Scenario 1

The first scenario is depicted in the following diagram:

Figure 1 - installation scenario 1

Network A will be connected via a 2-NIC Link Gateway called gw1.eg.net, Network B will be connected via a 1-NIC Link Gateway called gw2.eg.net and Network C will be connected via a 1-NIC gateway called gw3.eg.net.  The Link Gateways will automatically learn the routes to the networks behind each of the other Link Gateways via OSPF.  The peer-to-peer Link Network between the Link Gateways will be brokered by the Platform called vsp1.test.local.  The same Platform will be used to push configurations to each Link Gateway.

Note from the above diagram that gw1.eg.net will assign itself as the default gateway on Network A via DHCP to the DHCP clients on its internal network.  Link Gateway gw1.eg.net will automatically learn the routes to Network B and Network C via OSPF.

Note that the router which is the default gateway on Network B has RIP enabled, so enabling RIP on gw2.eg.net will allow the router to discover the routes to Network A and Network B via RIP.

Note that the router which is the default gateway on Network C does not have RIP enabled, so, assuming that RIP cannot be enabled on this router for some reason, the routes to Network A and Network B via gw3.eg.net will have to be added as static routes on this router.

Create the Link Network

The first step is to create the Link Network that will be used to connect the Link Gateways together.  Log into the Platform as an administrator and add a new network under Link Networking:

 

Note, that for security reasons, it is best to un-check Automatically approve devices enrolled to the network and click SAVE before enrolling any devices on the network.

Follow the Invitation URL

Follow the Invitation URL on each Link Gateway as described in Link Gateway Initial Configuration.

Check the Platform manual on how to approve devices (View Devices -> Unapproved Devices).

Click on the FQDN of the first Link Gateway to open up its configuration page:

Note that DNS-1 (10.2.5.5) and DNS-2 (10.2.5.6) under Internal Interface are DNS servers that can resolve the Platform address vsp1.test.local for the Link Gateway.  Under DHCP Configuration, DNS-1 and DNS-2 are the DNS server configurations that are pushed to the DHCP clients on the internal network.

Clicking the SAVE button will push this configuration to gw1.eg.net within about half a minute.

On the router which is the default gateway on Network C, add static routes to Network A and Network B via gw3.eg.net:

Pseudo-commands to add routes on Network C's default gateway
?
route add 172.16.1.0/24 via 192.168.11.5
route add 172.22.22.0/24 via 192.168.11.5

 

Test connectivity between networks

Check that PC A has been assigned a DHCP address by the Link Gateway gw1.eg.net.  If PC A has the incorrect address, then make sure that its network interface is set to receive its configuration via DHCP, disconnect the interface from the network and reconnect it.  After a few seconds, it should then receive its network configuration via DHCP from gw1.eg.net.

Check that PC A can access the Internet via gw1.eg.net.  Try to ping google.com.  If that does not work, then try to ping 8.8.8.8.  If the latter works, but not the former, then the issue is DNS resolution - check the DNS settings that were configured under DHCP Configuration on the Platform for gw1.eg.net.  If pinging 8.8.8.8 fails, then there is a networking issue that needs to be resolved.

Check that PC B and PC C have the correct addresses for their respective networks and resolve any configuration issues.

Each PC should then be able to ping the other PCs on the other networks.  If not, then check the firewall rules on each PC to allow ICMP echo requests and replies.

If these connectivity tests fail, then please refer to the Troubleshooting section below.

Troubleshooting

This troubleshooting section will refer to the setup in Scenario 1 above since it helps to have a real scenario in describing the troubleshooting process.

Link Network connectivity between Link Gateways

The first step is to check that the Link Gateways can communicate with each other over the Link Network.  This requires console access or secure shell (SSH) access into the Link Gateway:

SSH into Link Gateway
?
ssh -p 7712 net@link.gateway.address

Note than link.gateway.address can be the address of the internal interface, external interface or virtual interface on the Link Network.

You will be challenged for the password for user net that was set up during initial configuration of the Link Gateway.

Once logged into the Link Gateway, check that the TAP adapter is up by running ip a:

Listing interfaces
?
$ ip a

Note that the TAP adapter tap0 is up and has the IP address 10.1.1.1/24 bound to it, which agrees with what we see in the Devices page on the Platform:

 

If there is no TAP adapter listed, check that the Link Gateway can resolve the Platform address and communicate with it:

Ping Platform

Check that the peer process is running:

Check peer is running
?
$ pgrep peer
9755

The number is the process ID.  If there is no number listed, then the peer process is not running.  You can start it with:

Start peer
?
$ sudo systemctl start linkcore

Run pgrep peer again to check for the process ID.

Make sure that the Link Gateway has been approved on the Platform for the Link Network.

Once the TAP adapter is up, try pinging other nodes on the Link Network:

Ping other nodes
?
$ ping 10.1.1.2

If you can ping one node, but not the other, then chances are there is a problem on the latter Link Gateway.  Log into it and perform the same investigations.

Propagation of routes

Check that the routes to the other Link Gateway internal networks have been propagated via OSPF by running ip r:

Checking route propagation on gw1.eg.net
?
$ ip r
default via 10.22.5.1 dev enp0s8
10.1.1.0/24 dev tap0  proto kernel  scope link  src 10.1.1.1
10.22.5.0/24 dev enp0s8  proto kernel  scope link  src 10.22.5.87
169.254.0.0/16 dev enp0s3  scope link  metric 1002
169.254.0.0/16 dev enp0s8  scope link  metric 1003
172.16.1.0/24 dev enp0s3  proto kernel  scope link  src 172.16.1.1
172.22.22.0/24 via 10.1.1.2 dev tap0  proto zebra  metric 20
192.168.11.0/24 via 10.1.1.3 dev tap0  proto zebra  metric 20

The above show that the routes to the internal networks of gw2.eg.net (172.22.22.0/24) and gw3.eg.net (192.168.11.0/24) have been propagated to gw1.eg.net.  Notice that the gateways to these networks are the virtual addresses of the TAP adapters of the respective Link Gateways.

If a route to a network is not visible, then check that the virtual address of the other Link Gateway is pingable.  If not, then resolve its connectivity issues.  If it is, then wait for a minute for OSPF to propagate routes.  If the routes have not been propagated after a minute or two, then run tcpdump on the TAP interface to check for OSPF hello packets.
